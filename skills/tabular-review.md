# Tabular Review

The pattern Anthropic demoed in May 2026: a corporate counsel at Plutoco received a Box virtual deal room with 30 contracts as part of an M&A diligence process. One prompt invoked the tabular-review skill against the firm's diligence checklist. The output was an Excel spreadsheet — one row per contract, columns for change-of-control, assignment, term/termination, data rights, indemnification, and a severity flag (blocking / high / medium / low). Each cell carried either a clickable deep link back to the source contract or a "could not verify" flag.

That is what this skill builds. This file is the procedure.

---

## When to use this skill

- M&A buy-side diligence on a deal room.
- M&A sell-side preparing disclosure schedules from existing contracts.
- Discovery document review at scale.
- Deposition transcript review for thematic patterns.
- Regulatory filing review against a compliance checklist.
- Board-materials review (resolutions, minutes, certificates against a governance checklist).
- Any workflow where the deliverable is "a spreadsheet with these fields filled in for each of these documents."

## When *not* to use this skill

- You have one document. (Use targeted analysis or the long-document harness from `references/long-documents.md`.)
- You have many versions of *one* document. (Use the `version-diff` skill — different shape of problem.)
- The schema isn't known in advance. (Run a sample-and-discover pass first, then come back here once the schema is stable.)
- The work requires deep substantive analysis per document, not field-extraction with citations. (Tabular review is structured extraction; deep analysis belongs in a different shape.)

---

## Inputs

1. **Document source.** Either a connector pointing at a folder (Box, iManage, NetDocuments, SharePoint, Google Drive, local file system) or a drag-and-drop set. The skill prefers connector access for source-link generation.
2. **Schema.** A list of fields to extract. Can be passed inline ("change-of-control, assignment, term, indemnification, data rights"), pulled from a firm playbook, or generated by the skill from a sample document if no schema exists yet.
3. **Severity rubric.** Optional. If supplied, the skill flags severity per finding (blocking / high / medium / low). If absent, the skill produces extraction without severity classification.
4. **Output format.** Defaults to Excel `.xlsx`. Can be changed to CSV, Markdown table, or a Word table on request.

---

## Procedure

The skill executes in five phases. Each phase has a verification checkpoint.

### Phase 1 — Confirm the document set

List every document the skill will process. Show count, file types, and any unreadable items (scanned PDFs without OCR, password-protected files, broken connectors). Stop and ask before proceeding if any document can't be opened.

**Why this matters.** Silent failure on document loading is the single most common cause of bad tabular-review output. Confirm the universe before extracting.

### Phase 2 — Confirm the schema

Show the schema as a list with definitions. For each field, indicate:

- What it extracts (e.g. "term — the duration of the agreement, expressed in months").
- What format the cell will hold (free text / category / date / numeric / verbatim quote).
- Whether the field has a severity rubric.

Wait for confirmation. The cost of getting the schema wrong is high — re-running tabular review on a deal room of fifty contracts because column G was misdefined is annoying.

### Phase 3 — Sample run

Run the schema against three to five sample documents from the set. Produce a mini-spreadsheet with those rows filled in. Surface any extraction issues (ambiguous clauses, missing fields, schema mismatches). Stop and ask before processing the rest.

**Why this matters.** Harry's framing in the May demo: *"We don't want to run this tabular review on 200 or 500 documents if we don't test against maybe three to five documents first."* This is the failure-cheap iteration loop.

### Phase 4 — Full run

Process the full document set against the confirmed schema. For each document, populate every column. For each cell, either:

- Cite the source — a deep link back to the source document (preferred, when connector access supports it) or a passage reference (page / paragraph / clause).
- Flag "could not verify" — when the document doesn't contain the information, when the clause is ambiguous enough to require human reading, or when the extraction confidence is low.

**No fabricated cells.** If a field can't be extracted with citation, the cell is flagged, not guessed at. This is non-negotiable for legal use.

### Phase 5 — Verification pass

Re-read every "could not verify" cell. Re-read a sample of cited cells against the source to confirm the citation is accurate. Surface findings: any cells that should be flagged that aren't, any citations that are wrong.

Then summarize:

- Document count processed.
- Schema completeness (% of cells with citations, % flagged).
- Severity rollup (count of blocking / high / medium / low findings, if rubric was supplied).
- Top three findings by severity.
- Suggested next steps (which findings warrant attorney review, which can be cleared at lower review level, what additional documents to pull).

## Output format

The spreadsheet has two tabs by default:

### Tab 1 — Full review

| Column | Contents |
|---|---|
| A | Document name / identifier |
| B | Source link (clickable into the connector, where available) |
| C | Review date |
| D | Severity (if rubric supplied) |
| E onward | One column per schema field, with cell-level citation or "could not verify" flag |

### Tab 2 — Findings by category

Rolls up severity-flagged findings across the document set. For each finding, shows:

- Document(s) where the finding appears.
- Severity.
- Recommendation (e.g. "draft disclosure schedule exception," "negotiate change-of-control consent," "escalate to deal lead").
- Source citation.

This tab is what the deal lead reads. Tab 1 is what the lawyer reviews cell-by-cell during diligence.

## Cross-surface continuation

The output is an Excel file but the work doesn't end there. The same Claude session that produced the spreadsheet remembers the matter context. After tabular review:

- *"Open this in Excel, pull verbatim language from the cited clauses into a new column."* — Claude continues inside Excel via the add-in (see `references/microsoft-365.md`).
- *"Draft a Slack brief for the deal lead summarizing the blocking findings."* — Claude drafts via the Slack connector, never auto-sends.
- *"Generate a board-briefing slide from the severity rollup."* — Claude generates via the PowerPoint add-in.

This cross-surface continuation is the entire point of the May 2026 launch. Tabular review is the first leg of a relay, not the finish line.

## Privilege posture

If the documents are privileged (under common interest agreement, attorney-client work product, etc.), confirm three things before running:

1. The Claude tier supports your privilege posture (Team or Enterprise — see `references/privilege-layers.md`).
2. The connector authentication is firm-controlled, not personal.
3. The output destination (the Excel file's filesystem location) is inside the matter's privileged workspace.

The skill will insert a privilege header at the top of the spreadsheet if the practice profile indicates the user is a lawyer working on privileged material. Verify the header is present before sharing the file.

## Customization

Like every skill in this pack, tabular-review is a starting template. Common customizations:

- **Firm-specific severity rubric.** Replace the default blocking/high/medium/low with your firm's risk taxonomy.
- **Practice-area schemas.** Build named schemas (`commercial-msa`, `m-and-a-target-contracts`, `employment-separations`) and reference them by name in conversation.
- **Connector defaults.** Point the skill at a default folder for repeat workflows (e.g. "tabular-review on the current quarter's NDA folder").
- **Output styling.** Adjust column widths, conditional formatting, and tab layout to match your firm's standard diligence template.

See `references/skill-authoring.md` for the customization process. The cold-start interview (`references/cold-start-interview.md`) is where most of this lands, not in this file.

## Anti-patterns

- **Running on documents you haven't confirmed loaded.** Phase 1 exists for a reason.
- **Skipping the sample run.** Re-running tabular review on 50 contracts because column D was misdefined burns hours.
- **Treating "could not verify" cells as missing data.** They are *signals* that human review is needed on those specific cells. They are not bugs to be fixed by re-running with looser settings.
- **Auto-sending the output.** Drafts only. The deal lead reads, the lawyer approves, the human sends.
- **Using tabular review where deep analysis is needed.** If each document requires a paragraph of narrative analysis rather than field extraction, this isn't the right skill.
